## **1.1 AI運用方針の確定**

### **1.1.1 決定事項**

- LLMの分業（設計：ChatGPT、実装：Claude）は**不採用**
- Claude.ai（設計・壁打ち・理論議論） + Claude Code（実装）の**単一AI一貫体制**を採用

### **1.1.2 理由**

1. **設計と実装の境界が極めて薄い**: bank conflictのgcd計算、メモリ階層配置、atomicAdd選択の根拠など、「設計」と「実装」が一体不可分
2. **HPCの本質が計測→分析→最適化サイクル**: M1→M2→M3の過程で設計判断を修正する必要があり、AI間のラウンドトリップは非現実的
3. **コンテキスト断絶のリスク**: 2つのAI間で暗黙の設計意図が消失する
4. **設計の主導権はユーザー自身**: 必要なのは「設計してくれるAI」ではなく「壁打ち相手＋実装担当」

### **1.1.3 Claude Code協調環境**

| **ファイル** | **行数** | **役割** |
| --- | --- | --- |
| `CLAUDE.md` | 76行 | Claude Codeが毎セッション読む簡潔な指示書 |

**運用フロー**: Claude.aiでの設計議論 → ADRに記録 → GitHubにpush → Claude Codeがpull時に最新設計判断を反映

## **1.2 アーキテクト適性の確認**

ユーザーは「コードを書く人間ではなく構造で性能を出すアーキテクトタイプ」であることを確認。「キレイなアーキテクチャ」の基準は「制約と構造の対応関係が明確であること」（直交性＋必要十分性）。

| **評価** | **プロセッサ** | **理由** |
| --- | --- | --- |
| ✅ キレイ | SH2A | 16bit命令密度と組み込み制約の対応が完璧 |
| ✅ キレイ | Cell SPE | ストリーミング処理への潔い特化 |
| ✅ キレイ | Nios II | カスタム命令による問題適合型ISA |
| ✅ キレイ | Visconti | 画像認識パイプラインの各ステージに専用アクセラレータ |
| ❌ イケてない | RX | DSP/OoO/FPU全部載せで全て中途半端 |
| ❌ 複雑すぎ | ARM | 歴史的蓄積で直交性が崩壊 |

RHPC設計哲学に追加:
- 11: **「何を載せるか」ではなく「何を捨てるか」**
- 12: **道具に問題を合わせるのではなく、問題に道具を合わせる**

## **1.3 コンピュータアーキテクチャ考察**

### **1.3.1 Volta → Ampere 進化の系譜**

| **進化** | **内容** | **SCSCへの影響** |
| --- | --- | --- |
| 第3世代Tensor Core | TF32/BF16追加 | Phase 8のZMP制御で活用予定 |
| **Shared Memory hw atomic** | **ネイティブhw atomic** | **G(π)のatomicAddが低コスト** |
| 非同期データコピー（cp.async） | Global→Shared Memオーバーラップ | Prologue最適化に活用可能 |
| 構造化スパーシティ（2:4） | Tensor Core実効2倍 | Phase 8+で検討 |

### **1.3.2 CellとGPGPUの構造的類似性**

同じ物理法則（メモリウォール）に対する同じ構造的回答。「物理法則の制約を構造で回避する」= 「構造で性能を出す」。

| **対応** | **Cell** | **GPGPU** | **SCSC** |
| --- | --- | --- | --- |
| ローカルメモリ | SPE Local Store | Shared Memory | A,μをShared Memに配置 |
| ヘテロジニアス | PPE + SPE | CPU + GPU | ARM + Ampere + Teensy |
| データフロー | DMA + ダブルバッファ | Prologue/Epilogue | DRAM→SM→Reg→DRAM |

### **1.3.3 Condor Cluster**

PS3 1,760台、500 TFLOPS、200万ドル（同等スパコンの5-10%）、消費電力10%。教訓: 民生品活用、ドメイン特化、持続可能性リスク（OtherOS廃止で死亡）。

### **1.3.4 RoadRunner**

PowerXCell 8i 12,960個 + Opteron 6,480個、1.026 PFLOPS、1億ドル。PowerXCell 8iはFP64を8倍改善（12.8→102.4 GFLOPS）。3フェーズ段階的構築、3層ネットワーク分離（計算/IO/管理）、TriBlade最小構成単位。

### **1.3.5 RISC-V評価**

ISA品質は「教科書的RISC」で革新なし。真の価値は「オープンISA」。エクステンション乱立の懸念。ポテンシャルは「汎用ISA + ドメイン特化実装」。

### **1.3.6 Visconti評価**

画像認識パイプライン各ステージに専用アクセラレータ配置 = 「筋が良い」設計。2020年東芝リストラで中止。SCSCにはJetsonのISP+CUDAで代替可能。DNNの台頭で専用アクセラレータ群はDNN 1つに統合される時代に移行。

## **1.4 Jetsonクラスタ設計方針**

### **1.4.1 分類**

**分散メモリ型並列計算機**（グリッドではない）。データ依存関係あり、リアルタイム制約あり、集団通信パターンあり。

### **1.4.2 「第3の道」**

| **フェーズ** | **思想** | **内容** |
| --- | --- | --- |
| Phase 7 | RoadRunner | 単一ノード最適化。M1→M2→M3 |
| Phase 2初期 | Condor Cluster | 2台で動くものを最速で作る |
| Phase 2検証 | RoadRunner | 通信プロファイル、ボトルネック特定 |
| Phase 2拡張 | RoadRunner | 最小構成単位定義、階層的スケーリング |

### **1.4.3 ネットワーク設計**

チップ内レイテンシ（ns）とネットワークレイテンシ（μs〜ms）は1,000〜10,000倍差。warpスケジューリングでは隠蔽不能。対策: A/B行列を各ノードに複製、ポリシー評価のみ分散（G値交換40バイト）。FPGAは不採用。
